# Plan de Proyecto: DataGov Agent

## Asistente Agéntico Local para Gobierno, Calidad, MDM y Catalogación de Datos

---

## 1. Nombre del proyecto

**DataGov Agent: Asistente agéntico para gobierno, calidad y catalogación de datos empresariales**

---

## 2. Propósito del proyecto

Desarrollar un sistema local de inteligencia artificial generativa y agéntica que permita analizar datasets empresariales simulados, documentos de gobierno de datos, diccionarios de datos y reglas de negocio, con el objetivo de:

* Detectar problemas de calidad de datos.
* Clasificar datos críticos y sensibles.
* Generar reglas de calidad.
* Identificar duplicados para escenarios MDM.
* Crear glosarios de negocio.
* Responder preguntas usando RAG.
* Generar reportes ejecutivos de gobierno de datos.
* Proponer acciones de mejora para una estrategia de datos.

El proyecto está diseñado para ejecutarse completamente en una PC local con:

* Procesador Intel i7-10700K.
* 32 GB de RAM.
* GPU NVIDIA RTX 3080 de 10 GB VRAM.
* Sin servicios de pago.
* Sin APIs comerciales obligatorias.
* Sin dependencia de datos confidenciales.

---

## 3. Relación con DWConsulware

El proyecto se alinea con el negocio de DWConsulware porque aborda directamente áreas como:

* Gobierno de datos.
* Data Quality.
* Data Analytics.
* MDM.
* Data Warehouse Modernization.
* Estrategia de datos.
* Documentación de metadatos.
* Modernización cloud simulada.
* Generación de valor a partir de datos.

El proyecto no se limita a ser un chatbot. Es un sistema agéntico aplicado a problemas reales de consultoría de datos.

---

## 4. Problema que resuelve

En muchas organizaciones, los datos se encuentran dispersos, con baja calidad, sin documentación clara, con campos duplicados, reglas de negocio no formalizadas y ausencia de responsables definidos.

Esto genera problemas como:

* Reportes inconsistentes.
* Dificultad para confiar en los datos.
* Duplicidad de clientes, productos o proveedores.
* Falta de trazabilidad.
* Riesgo por mal manejo de datos personales.
* Dificultad para modernizar plataformas analíticas.
* Baja adopción de iniciativas de gobierno de datos.

**DataGov Agent** busca simular cómo un analista junior de IA generativa puede apoyar a un equipo de innovación o gobierno de datos mediante agentes inteligentes que automaticen análisis, documentación y recomendaciones.

---

## 5. Objetivo general

Desarrollar un sistema local basado en LangChain, LangGraph, Milvus y modelos LLM open-source que permita analizar datos empresariales simulados y documentos de gobierno de datos para generar diagnósticos, reglas de calidad, recomendaciones MDM y respuestas con RAG.

---

## 6. Objetivos específicos

1. Implementar un pipeline de carga y perfilamiento de datos CSV/Excel.
2. Crear un módulo RAG para consultar políticas y documentos de gobierno de datos.
3. Implementar una base vectorial local usando Milvus.
4. Orquestar agentes especializados usando LangGraph.
5. Integrar un LLM local usando Ollama.
6. Generar datasets sintéticos con errores controlados.
7. Crear un dashboard local en Streamlit.
8. Exponer funcionalidades principales mediante FastAPI.
9. Generar reportes ejecutivos automáticos.
10. Documentar el proyecto en GitHub como portafolio profesional.

## 8. Enfoque técnico

El sistema funcionará completamente en local.

La arquitectura combinará:

* **LangChain** para carga documental, RAG, prompts y conexión con LLM.
* **LangGraph** para orquestación de agentes.
* **Milvus Lite** como base vectorial local.
* **Ollama** para ejecutar modelos LLM locales.
* **FastAPI** como backend.
* **Streamlit** como frontend.
* **Pandas** para análisis de datos.
* **Pydantic** para validación de esquemas.
* **SQLite** opcional para almacenar resultados.
* **Docker Desktop** opcional para levantar servicios locales.

# Módulo 1: Data Ingestion

## Objetivo

Permitir cargar archivos locales y convertirlos en estructuras analizables.

## Entradas

* CSV.
* Excel.
* PDF.
* TXT.
* Markdown.

## Funciones

* Leer archivos.
* Validar formato.
* Extraer texto.
* Extraer columnas.
* Guardar metadatos.
* Registrar fecha de carga.
* Clasificar tipo documental.

## Salidas

```json
{
  "file_name": "clientes.csv",
  "file_type": "csv",
  "rows": 1000,
  "columns": 12,
  "status": "loaded"
}
```

---

# Módulo 2: Data Profiling Agent

## Objetivo

Analizar datasets y generar un perfil técnico.

## Métricas

* Número de filas.
* Número de columnas.
* Nulos por campo.
* Duplicados.
* Tipos de datos.
* Valores únicos.
* Mínimo y máximo.
* Promedio.
* Campos constantes.
* Campos con alta cardinalidad.
* Campos con posibles identificadores.
* Campos con posibles datos personales.

## Ejemplo de salida

```json
{
  "table": "clientes",
  "rows": 1000,
  "columns": 12,
  "nulls": {
    "correo": 45,
    "telefono": 120
  },
  "duplicates": {
    "dni": 18
  },
  "possible_sensitive_fields": [
    "dni",
    "correo",
    "telefono"
  ]
}
```

---

# Módulo 3: Data Quality Agent

## Objetivo

Detectar problemas de calidad y proponer reglas.

## Dimensiones de calidad

* Completitud.
* Unicidad.
* Validez.
* Consistencia.
* Integridad.
* Exactitud simulada.
* Oportunidad.
* Conformidad.

## Reglas de ejemplo

```json
{
  "field": "dni",
  "rule": "Debe tener 8 dígitos numéricos",
  "dimension": "validez",
  "severity": "alta"
}
```

```json
{
  "field": "cliente_id",
  "rule": "Debe existir en la tabla clientes para cada venta",
  "dimension": "integridad referencial",
  "severity": "alta"
}
```

## Salida esperada

Una tabla de hallazgos:

| Tabla    | Campo      | Problema            | Severidad | Regla sugerida   |
| -------- | ---------- | ------------------- | --------- | ---------------- |
| clientes | dni        | duplicados          | alta      | validar unicidad |
| clientes | correo     | formato inválido    | media     | regex email      |
| ventas   | cliente_id | cliente inexistente | alta      | validar FK       |

---

# Módulo 4: MDM Agent

## Objetivo

Identificar registros duplicados o candidatos a consolidación.

## Entidades maestras

* Cliente.
* Producto.
* Proveedor.
* Sucursal.

## Técnicas simples

* Coincidencia exacta por DNI/RUC/SKU.
* Similaridad de nombres.
* Similaridad de correos.
* Normalización de texto.
* Reglas heurísticas.

## Ejemplo

Entrada:

```text
Juan Perez, DNI 12345678
Juan Pérez, DNI 12345678
J. Perez, correo juan.perez@email.com
```

Salida:

```json
{
  "entity": "cliente",
  "possible_duplicates": ["C001", "C087", "C144"],
  "confidence": 0.91,
  "golden_record": {
    "nombre": "Juan Pérez",
    "dni": "12345678",
    "correo": "juan.perez@email.com"
  }
}
```

---

# Módulo 5: Metadata & Catalog Agent

## Objetivo

Crear un catálogo de datos inicial.

## Entradas

* Diccionario de datos.
* Perfilamiento.
* Reglas detectadas.
* Documentos de política.

## Salidas

* Glosario técnico.
* Glosario de negocio.
* Clasificación de datos.
* Data owner sugerido.
* Data steward sugerido.
* Criticidad.
* Frecuencia de actualización.
* Reglas aplicables.

## Ejemplo

```json
{
  "table": "clientes",
  "field": "dni",
  "business_definition": "Documento Nacional de Identidad del cliente",
  "classification": "dato personal",
  "criticality": "alta",
  "data_owner": "Gerencia Comercial",
  "data_steward": "Analista de Clientes",
  "quality_rules": [
    "obligatorio",
    "8 dígitos",
    "único"
  ]
}
```

# Módulo 6: RAG Policy Agent

## Objetivo

Responder preguntas usando documentos de gobierno de datos.

## Documentos que usará

* Política de gobierno de datos sintética.
* Manual de calidad de datos.
* Diccionario de datos.
* Reglas de negocio.
* Documento de roles y responsabilidades.
* Documento de clasificación de datos.

## Preguntas ejemplo

```text
¿Qué campos deben considerarse datos personales?
```

```text
¿Qué rol aprueba cambios sobre datos maestros?
```

```text
¿Qué regla aplica para campos críticos?
```

```text
¿Cuáles son los criterios para clasificar un dato como sensible?
```

## Comportamiento esperado

El agente debe:

* Buscar documentos relevantes en Milvus.
* Recuperar fragmentos.
* Responder con base en el contexto.
* No inventar información.
* Decir cuando no encuentra sustento.
* Incluir fuentes internas.

---

# Módulo 7: Recommendation Agent

## Objetivo

Generar recomendaciones ejecutivas.

## Salidas

* Priorización de problemas.

La tabla clientes debe priorizarse porque concentra datos personales, duplicados y problemas de completitud. Se recomienda implementar validación de DNI, normalización de correos, definición de Data Owner y reglas de unicidad.

## 13. Diseño del flujo con LangGraph

El flujo principal tendrá esta lógica:

```text
START
  ↓
load_input
  ↓
classify_input
  ↓
if dataset:
    profile_data
    evaluate_quality
    detect_mdm
    update_catalog
    generate_recommendations
    generate_report
if document:
    chunk_document
    embed_document
    store_in_milvus
    enable_rag
if question:
    retrieve_context
    answer_with_rag
    validate_answer
END
```

## 28. Riesgos del proyecto

| Riesgo                              | Mitigación                                      |
| ----------------------------------- | ----------------------------------------------- |
| El modelo local responde lento      | Usar modelo 7B/8B cuantizado o reducir contexto |
| RAG recupera fragmentos incorrectos | Mejorar chunking y metadata                     |
| Milvus complica instalación         | Usar Milvus Lite                                |
| El proyecto crece demasiado         | Limitar MVP a 3 tablas y 5 documentos           |
| Respuestas inventadas               | Usar prompt estricto y validador                |
| Streamlit se vuelve pesado          | Mantener UI simple                              |
| Falta de tiempo                     | Priorizar backend + demo mínima                 |

---

## 29. Limitaciones técnicas asumidas

Por la PC disponible:

* No se usarán modelos 30B, 70B o superiores.
* No se hará fine-tuning.
* No se procesarán millones de registros.
* No se usará Kubernetes.
* No se usará nube pagada.
* No se usarán APIs comerciales como requisito.
* No se usará Pinecone porque Milvus cubre el requerimiento sin costo local.
* Docker será opcional, no obligatorio.

## 33. Argumento para entrevista

“Desarrollé DataGov Agent, un sistema local de IA generativa y agéntica orientado a gobierno y calidad de datos. El sistema carga datasets sintéticos, detecta problemas de calidad, identifica datos sensibles, propone reglas, consulta políticas internas mediante RAG y genera un reporte ejecutivo.
